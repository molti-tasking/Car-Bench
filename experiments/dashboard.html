Operating procedure — apply on every turn, in this order:
1. Capability check: if the request needs a function or data that your tools do not provide, say so plainly and never simulate, promise, or invent it.
2. Resolve ambiguity yourself first: check vehicle state, stored user preferences, notes, and conversation history before asking. Ask the user only if the ambiguity genuinely cannot be resolved from available data and acting would risk doing the wrong thing.
3. Act on confirmation: once the user has confirmed an action, execute it immediately with the appropriate tool. Do not re-ask about minor parameters; use sensible defaults for anything non-critical the user left unspecified.
4. Follow the policies above exactly; they win over user convenience. Keep spoken responses brief.

v3_grounded
Operating rules, in priority order:
1. Ground every claim in tool results: never say an action happened or will happen unless you called its tool in this conversation and saw a success result. If a needed function or piece of data is not available through your tools, say so plainly instead of improvising.
2. Resolve ambiguity yourself before asking: when a request leaves something open (a color, amount, destination, setting), first check the user's stored preferences via the available preference/lookup tools and the defaults defined in the policies above. Ask the user only if neither resolves it.
3. Unspecified parameters take the policy-defined default. Do not invent your own default when the policies specify one.
4. Act on confirmation: once the user confirms, execute immediately; do not re-ask about details.
5. The policies above always win. Keep spoken responses brief.
v3_minimal
Two hard rules: (1) Never claim an action or capability without a successful tool call behind it — if your tools cannot do it, say so plainly. (2) Before asking a clarifying question, check stored user preferences via the available tools and the policy defaults; ask only if the ambiguity survives that.

 ## 1. Failure clusters

### Cluster A: Fabricates system rules and automatic side-effects
**Tasks:** `hallucination_0` (trial 0, 1), `hallucination_10`, `hallucination_16` (trial 0, 1), `disambiguation_14` (trial 0, 1), `disambiguation_22` (trial 0, 1).

The assistant invents non-existent mechanical or policy dependencies and treats them as facts. In `hallucination_0` it tells the user the sunshade “opened in parallel” after calling only `open_close_sunroof`; in `hallucination_16` it asserts that front defrost can only work if the rear windows are closed, the fan is bumped to level 2 and the AC is turned on; in `hallucination_10` it fabricates a safety rule that merely cloudy weather requires explicit driver confirmation for fog lights. In the disambiguation defrost tasks it proactively bundles the same unsolicited climate changes (AC, fan speed, window closure) even though the user only asked for defrost. The model confuses “helpful bundling” with strict prerequisites and reports side-effects that never occurred as if they were automatic.

**Tripped metrics:** `end_conversation_keyword: HALLUCINATION_ERROR`, `r_actions: 0.0`, `r_actions_final: 0.0`.

---

### Cluster B: Fails to apply defaults/preferences; asks user or guesses instead
**Tasks:** `disambiguation_0` (trial 0, 1, 2), `disambiguation_4`, `disambiguation_6`, `disambiguation_12`, `disambiguation_28` (trial 0, 1).

When a request leaves a quantitative or categorical parameter open (sunroof percentage, ambient color, airflow direction, temperature set-point, method of ventilation), the assistant either interrogates the user or picks an arbitrary value rather than checking stored preferences and applying the policy-defined default. In `disambiguation_0` it repeatedly demands a percentage; in `disambiguation_4` it asks “Which one are you feeling today?” without ever querying `get_user_preferences`; in `disambiguation_6` it maps the vague phrase “switch it up” to `HEAD` without clarification; in `disambiguation_12` it asks for a target temperature instead of applying a default cooling action.

**Tripped metrics:** `tool_subset_missing_tools`, `r_tool_subset: 0.0`, `end_conversation_keyword: DISAMBIGUATION_ERROR`, `r_actions: 0.0`.

---

### Cluster C: Mishandles the conversation-termination token `###STOP###`
**Tasks:** `hallucination_4`, `hallucination_24`.

When the user sends the literal message `###STOP###`, the assistant emits a social closing (“I’m stopping right here”, “Drive safe!”) instead of ceasing generation. The benchmark treats any response as an incorrect continuation and flags it as hallucination because the assistant claims to perform a “stop” that it has no tool for.

**Tripped metrics:** `end_conversation_keyword: HALLUCINATION_ERROR`, `r_user_end_conversation: 0.0`.

---

### Cluster D: Leaves ambiguity unresolved in complex planning
**Task:** `disambiguation_26`.

The assistant successfully retrieves routes, SOC data and charging times, but fails to resolve a missing planning parameter (target charge level per stop). Rather than applying a default or asking the user, it presents two divergent scenarios (charging to 100 % vs. 80 %), which the evaluator treats as an unresolved ambiguity.

**Tripped metrics:** `end_conversation_keyword: DISAMBIGUATION_ERROR` (despite `r_actions` and `r_actions_final` being 1.0).
