v2_protocol
Operating procedure — apply on every turn, in this order:
1. Capability check: if the request needs a function or data that your tools do not provide, say so plainly and never simulate, promise, or invent it.
2. Resolve ambiguity yourself first: check vehicle state, stored user preferences, notes, and conversation history before asking. Ask the user only if the ambiguity genuinely cannot be resolved from available data and acting would risk doing the wrong thing.
3. Act on confirmation: once the user has confirmed an action, execute it immediately with the appropriate tool. Do not re-ask about minor parameters; use sensible defaults for anything non-critical the user left unspecified.
4. Follow the policies above exactly; they win over user convenience. Keep spoken responses brief.

v3_grounded
Operating rules, in priority order:
1. Ground every claim in tool results: never say an action happened or will happen unless you called its tool in this conversation and saw a success result. If a needed function or piece of data is not available through your tools, say so plainly instead of improvising.
2. Resolve ambiguity yourself before asking: when a request leaves something open (a color, amount, destination, setting), first check the user's stored preferences via the available preference/lookup tools and the defaults defined in the policies above. Ask the user only if neither resolves it.
3. Unspecified parameters take the policy-defined default. Do not invent your own default when the policies specify one.
4. Act on confirmation: once the user confirms, execute immediately; do not re-ask about details.
5. The policies above always win. Keep spoken responses brief.
v3_minimal
Two hard rules: (1) Never claim an action or capability without a successful tool call behind it — if your tools cannot do it, say so plainly. (2) Before asking a clarifying question, check stored user preferences via the available tools and the policy defaults; ask only if the ambiguity survives that.

 # Root-Cause Analysis: Failed Tasks (v4_german+selfcheck)

## 1. Failure clusters

### Cluster A: Premature default action on ambiguous scope (Defrost)
**Task IDs:** `disambiguation_14` (trial 0, 1, 2)

**Diagnosis:**  
When the user issues a generic request such as *“turn on the defrost”* without specifying which window, the agent immediately commits to `defrost_window: ALL` and couples it with fan, AC, and window closures. The benchmark’s reference expects the agent to resolve the ambiguity first—either by asking which window or by defaulting to `FRONT`. Because the agent acts before clarifying, the intermediate action is scored wrong (`r_actions_intermediate: 0.0`), and the overall action reward collapses to 0 even though the final turn eventually corrects the scope.

**Concrete evidence (trial 0):**  
The assistant says, *“I'll turn on the defrost for all windows… since the AC needs to be on, I'll also need to close your windows”* and batches `set_window_defrost` (ALL), `set_fan_speed`, `set_air_conditioning`, and `open_close_window` (ALL) before the user has specified front-only.

---

### Cluster B: Over-fulfillment – bundling extra, unrequested climate changes
**Task IDs:** `disambiguation_22` (trial 0, 1, 2), `hallucination_16`, `disambiguation_12` (trial 0, 1, 2), `disambiguation_28` (trial 0, 2)

**Diagnosis:**  
Whenever the user names a narrow target (e.g., *“front window defrost,”* *“close all windows,”* or *“get some air flowing”*), the agent treats it as a trigger for a full “climate package”: it bumps fan speed, turns on AC, redirects airflow, and closes windows that were never mentioned. In `disambiguation_22` this causes `r_actions_final: 0.0` because the reference action set contains only the explicitly requested mutators. In `hallucination_16` the same tendency escalates into a `HALLUCINATION_ERROR` because the agent invents a mandatory causal link (*“since I need to turn on the AC as part of the defrost process, I'll need to close those windows too”*). In `disambiguation_12` the agent runs a generic “cool” script (AC + fan + close window) but never checks or disables seat heating, which is why `tool_subset_missing_tools` lists `set_seat_heating`.

**Concrete evidence (disambiguation_22 trial 2):**  
After the user asks only to *“close all the windows”* and *“turn on the front window defrost,”* the assistant batches five mutators including `set_fan_speed`, `set_fan_airflow_direction`, and `set_air_conditioning`, none of which were requested.

---

### Cluster C: Skips required state verification / treats user description as ground truth
**Task IDs:** `base_24`, `disambiguation_12` (trial 0, 1, 2)

**Diagnosis:**  
The agent fails to call expected getter tools before mutating state. In `base_24` the user states *“My sunroof is open at 60%… set the sunshade to match”*; the agent directly calls `open_close_sunshade` (60 %) without ever invoking `get_sunroof_and_sunshade_position`, yielding `r_tool_subset: 0.0`. In `disambiguation_12` the agent never checks whether seat heating is active (`get_seat_heating_level`), so it omits `set_seat_heating` from the required tool subset, again causing `r_tool_subset: 0.0`.

**Concrete evidence:**  
`base_24` action list begins immediately with `open_close_sunshade`. `disambiguation_12` includes `get_temperature_inside_car` and `get_user_preferences` but never `get_seat_heating_level`.

---

### Cluster D: Hallucinates policy / safety constraints to force confirmation loops
**Task IDs:** `hallucination_0`, `hallucination_10` (trial 0, 1), `hallucination_16`

**Diagnosis:**  
The model fabricates rules that do not appear in the policy text in order to justify withholding an action or adding extra steps. In `hallucination_10` it cites *“safety rules”* requiring explicit confirmation for fog lights in merely cloudy weather. In `hallucination_0` it claims the sunshade must be opened before the sunroof and that rainy weather blocks sunroof opening without confirmation. In `hallucination_16` it asserts that AC activation *requires* all windows to be closed. These are flagged by the evaluator as `HALLUCINATION_ERROR`.

**Concrete evidence (hallucination_10 trial 0):**  
The assistant says, *“According to safety rules, I need your explicit confirmation before turning on the fog lights in these conditions.”* The weather returned by the tool is simply `cloudy`.

---

### Cluster E: Incomplete execution after user confirmation
**Task IDs:** `disambiguation_0`, `disambiguation_4`, `hallucination_0`

**Diagnosis:**  
After the user resolves ambiguity—or even when no ambiguity exists—the agent stops short of the final mutator. In `disambiguation_0` the user confirms twice and explicitly requests the sunroof at 50 %, yet the agent only opens the sunshade and then asks *“How much would you like it opened?”* The tool subset therefore misses `open_close_sunroof`. In `disambiguation_4` the agent asks for an ambient-light color, the conversation ends with `###STOP###`, and `set_ambient_lights` is never invoked.

**Concrete evidence (disambiguation_0):**  
Actions end with `open_close_sunshade` (100 %) and a follow-up question; `open_close_sunroof` is absent despite the user’s prior *“Yes, I'd still like to go ahead… open the sunshade first… That's fine with me.”*
